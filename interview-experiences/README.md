#### 高频问题

- AI 报警服务怎么实现，AI 输入了什么数据、输出了什么结果。
- 加入 AI 之后业务指标有什么变化，准确性如何保障。
- 遇到发版、节假日等突发情况时，AI 预测失效怎么补救。
- RAG 处理过什么文件，不同类型文件如何解析。
- 遇到几个 G、几千页、图文混杂的大文件时如何实现。
- RAG 的更新频率、更新方式是手动还是自动。
- RAG 在实际业务中解决了什么问题，优化效果如何衡量。
- LangChain 在 AI 助手里的具体用法。
- AI 助手如何接入办公 OA，如何区分多条调用链路和写链路。
- MCP Server 提供了哪些工具，如何保障分析正确性。
- 对当前 AI Agent 技术发展、RAG、Skills 的看法。

#### 面试启发

- 面试官非常重视“项目闭环”，不是只问做了什么，而是追问输入、输出、指标、异常和补救机制。
- RAG 相关问题会直接落到工程细节，例如文件解析、更新策略、海量文档处理和业务效果评估。
- Agent / MCP 问题通常会继续追问到工具边界、调用链路和结果可信度。

### 2. 昨天的面试情况

- 来源：LINUX DO
- 日期：2025-05-14
- 链接：https://linux.do/t/topic/648771

#### 高频问题

- 给博物馆做 RAG 问答，如果用户问“有哪些唐代文物”，检索效果不好时怎么处理。
- 多轮对话里，前两问分别查询不同专业课程，第三问要求比较两个专业就业前景，此时如何做上下文理解和检索。
- 如果不能调用大模型做 Query Rewrite，如何在保证时延的前提下处理指代、比较、改写和检索。

#### 面试启发

- 工业级 RAG 往往不是“向量检索 + 大模型”这么简单，而是包含 FAQ 拦截、意图识别、实体提取、对话状态管理、业务规则和多路召回。
- 面试官会关注 latency 和 precision 的权衡，所以不能只说“再调一次大模型重写 Query”。
- 对于“唐代文物”这类问题，元数据过滤、标签体系、结构化字段、FAQ/Base Case 都是常见思路。
- 对于“比较前文两个实体”这类问题，需要上下文状态维护、指代消解、模板化改写或轻量模型，而不是只会把历史对话整段丢给 LLM。

#### 推荐回看章节

- `07-memory-and-rag`
- `08-tool-use`
- `10-real-world-cases`

## 使用建议

1. 先通读面经里的提问方式，观察面试官是如何从“做过项目”追问到“是否真正上线过”的。
2. 再回到题库中按章节补知识点，不要只记结论，要准备输入、输出、指标、异常、降级、成本和时延这些工程化回答。
3. 如果后续继续收录面经，优先补充来源、日期、原始链接、核心问题和对应复习章节。
